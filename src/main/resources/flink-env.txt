**项目背景：**

**技术栈：**
DataX、HDFS、Hive、DolphinScheduler、PostgreSQL、Superset/SugarBI

**个人职责：**

1. 参与离线数仓分层设计，按照 ODS、DWD、DWS、ADS 结构建设订单、商品、用户、地区等主题数据表。
2. 基于 DataX 将 MySQL/PostgreSQL 业务库数据同步至 HDFS/Hive，完成订单表、订单明细表、商品表、用户表、地区表等数据采集。
3. 负责 DWD 层订单明细事实表开发，对订单状态、支付时间、退款金额、取消订单、重复订单等字段进行清洗和标准化处理。
4. 负责 DWS 层主题汇总表开发，按用户、商品、地区、日期等维度汇总订单数、支付金额、支付用户数、退款金额等指标。

## 项目经历三：用户行为实时规则计算与营销触达候选事件建设

**项目背景：**
业务希望基于用户行为进行实时运营触达，例如用户创建待付款订单后一段时间仍未支付时，输出催付或优惠券候选事件；同时需要考虑用户触达频控，避免同一用户短时间内被重复推送，降低用户打扰和营销成本。

**技术栈：**
Kafka、Flink DataStream API、Flink CDC、Flink State、Timer、Checkpoint、Redis、Doris/ClickHouse

**个人职责：**

1. 参与用户行为实时规则链路设计，消费订单创建、支付成功、订单关闭等实时事件，按照用户和订单维度进行状态计算。
2. 基于 Flink DataStream API 使用 KeyedProcessFunction / KeyedCoProcessFunction 处理多条实时事件流。
3. 使用 ValueState 保存用户待付款订单状态，当订单进入待付款状态时注册 Timer，到达指定时间后检查订单是否仍未支付。
4. 当支付成功事件到达时，清理对应订单状态并删除 Timer，避免用户已支付后仍被误触达。
5. 当订单关闭或取消事件到达时，清理状态并终止该订单的触达判断，避免无效触达。
6. 基于用户维度设计触达频控逻辑，限制同一用户同一规则在固定周期内重复触发，避免用户反复薅券或被频繁打扰。
7. 通过 MapState/ListState 保存用户历史触达记录和候选触发记录，用于判断用户是否存在频繁待付款不支付等异常行为。
8. Flink 仅输出营销触达候选事件到 Kafka，最终发券由营销系统结合订单状态、券库存、用户资格、Redis 幂等 Key 等进行二次校验。
9. 将触达候选事件和触达结果写入 Doris/ClickHouse，用于运营看板分析和后续规则效果评估。

**项目成果：**

1. 实现待付款订单超时未支付的实时识别，为营销系统提供催付/优惠券候选事件。
2. 通过 State + Timer 机制实现延迟判断，避免使用离线任务造成触达不及时。
3. 通过用户维度频控、状态 TTL、幂等 Key、订单状态二次校验等机制，降低重复触达和误触达风险。
4. 沉淀用户行为规则计算链路，为后续扩展加购未支付、重复加购、浏览未购买等实时营销场景提供基础。

## 可写在简历里的重点 SQL / 数据问题

1. **同比环比：** 先按日期或月份聚合指标，再通过 `lag` 或日期自关联获取上一周期或去年同期数据，计算增长额和增长率。
2. **每个用户最近一笔订单：** 使用 `row_number() over(partition by user_id order by pay_time desc)` 获取用户最近支付订单。
3. **最近 7 天滚动活跃用户：** 通过日期维表关联过去 7 天活跃明细，使用 `count(distinct user_id)` 统计滚动活跃，避免直接累加 DAU 导致重复计算。
4. **用户生命周期分层：** 基于注册时间、最近活跃时间、最近支付时间、累计支付金额等字段，将用户划分为新用户、活跃用户、沉默用户、流失用户、回流用户、高价值用户等。
5. **拉链表取有效状态：** 使用 `start_date <= 业务日期 and end_date >= 业务日期` 关联维度表，保证事实数据关联到当时有效的历史维度状态。
6. **Join 后金额膨胀排查：** 对比 Join 前后行数、订单去重数、金额汇总结果，检查维表主键是否重复、一对多关联是否合理，并通过维表去重、先聚合后关联、明确多值分摊口径等方式解决。
